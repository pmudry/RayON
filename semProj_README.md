## 1. Project Overview

> **RayON** is an interactive path tracing engine written in C++ and CUDA. It features a progressive GPU renderer, YAML scene management, and real-time visualization tools.

**Key Features & Capabilities:**

*   **Hybrid Rendering Architecture:**
    *   **CPU Backends:** Sequential and Parallel (OpenMP) implementations serve as "ground truth" references for validating light transport algorithms.
    *   **CUDA GPU Backend:** A highly optimized, pointer-based rendering pipeline that leverages massive parallelism for speed.
*   **Interactive Progressive Rendering:**
    *   A real-time mode (Method 3) utilizing SDL2 and ImGui.
    *   Features a "progressive" accumulation buffer: moving the camera provides instant, low-sample feedback, which progressively refines to a noise-free image when stationary.
    *   Dynamic controls for samples, depth, and target FPS.
*   **Scene Management:**
    *   **YAML-based Scene Description:** Scenes are defined in flexible `.yaml` files, supporting custom materials, geometry, and camera settings.
    *   **Asset Support:** capable of loading complex Wavefront OBJ meshes (e.g., Sponza Atrium, Conference Room) with associated material libraries (`.mtl`).
*   **Performance & Benchmarking:**
    *   **Automated Benchmark Suite:** A dedicated Python-based framework (`scripts/run_benchmarks.py`) to execute reproducible performance tests across various scenes and resolutions.

**Current Status:**
Although still a prototype for large scene, the project could be used as intended : a demo tool for the ISC cursus.

## 2. User Manual

### Prerequisites
*   **Compiler:** C++17 compliant (GCC/Clang)
*   **CUDA Toolkit:** Required for GPU rendering (NVCC)
*   **CMake:** 3.8 or newer
*   **SDL2:** Required for the interactive viewer (`sudo apt install libsdl2-dev` on Ubuntu)

### Building the Project
```bash
mkdir build
cd build
cmake .. -DCMAKE_BUILD_TYPE=Release
make -j
```

### Running RayON

> **Note:** For a complete list of command-line options, see [CLI_ARGUMENTS.md](CLI_ARGUMENTS.md).

**1. Benchmark / Offline Rendering:**
Render a scene to an image file.
```bash
./rayon --scene resources/scenes/benchmark/cornell_box.yaml -s 100 -r 1080
```
*   `-s`: Samples per pixel.
*   `-r`: Vertical resolution (e.g., 720, 1080).

**2. Interactive Mode:**
Explore the scene in real-time.
```bash
./rayon -m 3 --scene resources/scenes/benchmark/fireplace_room.yaml
```
*   `-m 3`: Selects the "CUDA Progressive Interactive" mode.

**Interactive Controls:**

*   **Mouse:**
    *   **LMB (Left Click + Drag):** Rotate Camera
    *   **RMB (Right Click + Drag):** Pan Camera
    *   **Scroll Wheel:** Zoom In/Out

*   **Keyboard Shortcuts:**
    *   **W/A/S/D:** Move Camera (Forward/Left/Back/Right)
    *   **Space:** Toggle Automatic Sample Accumulation
    *   **O:** Toggle Auto-Orbit Mode
    *   **N / P:** Switch to Next / Previous Scene
    *   **H:** Toggle UI Visibility
    *   **C:** Collapse/Expand All UI Headers
    *   **Esc:** Exit Application

### Benchmarking
> **Note:** For a comprehensive guide on running and customizing benchmarks, refer to the [RayON Benchmarking Guide (README_BENCHMARK.md)](README_BENCHMARK.md).

## 3. Developer Guide

### Architecture
The project is organized in `src/rayon/` with key components:
*   **`RenderCoordinator`**: Manages the render loop, camera, scene, and renderer.
*   **`IRenderer`**: Interface for all rendering backends (CPU, CUDA).
*   **`SceneFactory` & `YamlSceneLoader`**: (located in `src/rayon/scenes/`) Handle scene parsing and construction.

### Adding a New Feature
1.  **New Material:**
    *   Add the type to `MaterialType` in `material.hpp`.
    *   Implement the scattering logic in `cuda_raytracer.cu` (device) and `material.hpp` (host).
    *   Update `yaml_scene_loader.cc` to parse the new material properties.

2.  **New Shape:**
    *   Define the struct in `hittable.hpp`.
    *   Implement the intersection logic in `cuda_raytracer.cu`.
    *   Add parsing support in `YamlSceneLoader`.

## 4. Testing

### System Tests (Benchmarks)
We use a suite of benchmark scenes to ensure performance stability and correctness.
*   **Configs:** The benchmark configurations are YAML files, typically found in `benchmark_configs/`, and can be automatically generated by `scripts/generate_benchmark_configs.py`. These files define specific scenes and rendering parameters for testing.
*   **Runner:** `scripts/run_benchmarks.py` executes these configs and generates a CSV summary.
*   **Outputs:** The script saves outputs to `benchmark_results/` for manual inspection against reference images.

## 5. Project Timeline & Contributions

This section outlines the chronological development and key contributions made during the semester project.

*   **Phase 1: ImGui Integration & Interactive UI**
    *   **Modern Interface:** Replaced the old text overlay with a modern, graphical user interface (**Dear ImGui**) that sits on top of the rendered image.

*   **Phase 2: Asset Loading & Scene Management**
    *   **3D Model Support:** Integrated a tool (**`tinyobjloader`**) to read standard 3D model files (`.obj`), allowing the engine to display complex objects like statues or rooms instead of just simple spheres.
    *   **Triangle Geometry:** Implemented a system to handle "meshes" (objects made of thousands of tiny triangles) efficiently.
    *   **Scenes and materials properties:** Added the ability to read scene properties directly from scene YAML files.

*   **Phase 3: Benchmarking Framework**
    *   **Automated Testing:** Developed a scripted system (using Python) to run performance tests automatically.
    *   **Reproducible Results:** Created a tool that generates standard test scenarios, ensuring that performance measurements are consistent and fair.

*   **Phase 4: Optimization & Refactoring**
    >   **Guided Optimization:** The previously developed benchmarking suite and profiling tools (like `nsight-compute`) were crucial. They allowed for precise identification of performance bottlenecks and objective verification of the impact of each optimization.
    *   **Math Enhancements:** Reduced instruction count by utilizing hardware Special Function Units (SFUs) for trigonometric and root operations, and optimized vector math to minimize register pressure.
    *   **Warp-Level Primitives:** Replaced global atomic operations with warp-shuffle intrinsics (`__shfl_down_sync`) to aggregate partial results within a thread block before committing to global memory, significantly reducing contention.
    *   **Traversal Optimization:** Improved BVH traversal logic by ordering child node processing based on ray direction (pushing the farther node onto the stack first), ensuring closer intersections are processed earlier for better early-out.
    *   **Loop Invariant Code Motion:** Moved constant ray properties (such as inverse direction for slab tests) out of the traversal loop, precomputing them once to reduce redundant arithmetic operations during intersection tests.
